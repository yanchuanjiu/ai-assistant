## 已知注意事项

- ✅ 已解决（v0.8.12）：用户6次请求创建3个项目目录均失败。根因：服务于10:12启动，当时代码为v0.8.7之前版本，缺少 `feishu_project_setup` 工具和 space_id 识别修复。v0.8.12 已修复，服务重启后生效
- **代码生效需重启**：每次 commit 后必须重启服务，否则 Python 内存中仍运行旧代码。每次自我改进完成后检查：若 `logs/service.pid` 对应进程启动时间早于最新 commit，必须重启
- **parent_wiki_token 不能传 space_id**：飞书空间ID（纯数字如 `7618158120166034630`）不是有效的 wiki token，v0.8.8/v0.8.12 已修复，但 LLM 仍可能传参错误，需 `_is_space_level_token()` 兜底
- **响应延迟极高（⚠️ 未解决）**：69条交互中97%超过15秒，p50=112s，p95=658s，max=2975s。用户多次明确投诉（"太慢"），根因：主飞书线程平均input token 82K（max 203K），多轮工具调用链积累。v0.8.27 并发框架已部分改善，但延迟问题根本上来自 token 过重，需要更激进的上下文截断
- **token 消耗极重（⚠️ 更新）**：主飞书线程 `feishu:oc_d2b5bb8b3ee37fdc61aebaa9d988192b` 调用 378 次，avg input 82K，max 203K，**严重超过阈值**。HISTORY_TOOL_CONTENT_LIMIT=300 的效果仍不足
- **飞书操作从首页而非空间根节点开始（✅ v0.8.28 修复）**：LLM 调用 `feishu_wiki_page(list_children)` 时，因为代码要求必须传 parent_wiki_token，导致 LLM 用 FEISHU_WIKI_CONTEXT_PAGE（AI自己的页面）作起点，看不到用户的项目目录，还在错误位置创建页面。v0.8.28 修复：list_children 不传 parent 时默认从空间根节点开始
- **多任务上下文污染（v0.8.20 修复）**：用户在同一会话中处理多个不相关任务时，历史工具结果（A项目的飞书页面内容）会污染后续任务（B项目）的上下文。v0.8.20 在 `_trim_to_user_turns` 中增加历史轮 ToolMessage 内容截断（HISTORY_TOOL_CONTENT_LIMIT=300），同时在 system.md 中增加跨任务隔离规则。
- **admin-http 端口冲突（已知稳定问题）**：8080 端口重复占用导致 admin 线程持续崩溃，截至2026-03-21共14次。属已知稳定问题，不影响主服务，每天仅提醒一次。
- **错误响应率**：69条交互中13条包含"失败/错误/出错"（18.8%），主要为飞书API操作失败
